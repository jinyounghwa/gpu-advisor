## 1. 프로젝트 개요

GPU Advisor는 **AlphaZero/MuZero 아키텍처**를 GPU 시장의 구매 의사결정 도메인에 적용한 개인용 AI 에이전트 시스템이다.
단순한 가격 예측기가 아니라, "지금 사야 하는가, 기다려야 하는가"라는 **행동 결정(action decision)** 에 최적화되어 있다.

**핵심 구성 요소:**

| 계층 | 구성 |
|------|------|
| 데이터 수집 | 다나와 GPU 가격 + 환율 + GPU 뉴스 (매일 자정 자동 수집) |
| 특징 엔지니어링 | 가격·환율·뉴스·시장·기술지표 → 256차원 벡터 |
| AI 모델 | Representation h(s) + **World Model g(s,a)** + Prediction f(s) + **Action Model a(z)** (총 ~19M 파라미터) |
| 계획 | MCTS (50 simulations × 5 rollout steps) |
| 파이프라인 | 자동 재학습 → 품질 게이트 → 릴리즈 판정 → 한국어 보고서 생성 |
| 서비스 | FastAPI REST API + Next.js 프론트엔드 |

## 2. 장점

### 2-1. 도메인 적용의 독창성 — "가격 예측"이 아닌 "의사결정 최적화"

대부분의 주가·가격 예측 프로젝트는 회귀(regression) 문제로 접근하여 "내일 가격이 얼마일 것인가"를 예측한다.
GPU Advisor는 이 프레임을 거부하고 **강화학습의 행동 최적화** 문제로 재정의했다.

- 예측 오차를 줄이는 것이 목표가 아니라, **기댓 보상을 최대화하는 행동 정책**을 학습
- BUY_NOW의 정확도가 60%라도 적절한 타이밍에만 실행하면 나쁜 예측기보다 좋은 에이전트가 될 수 있음
- 이는 "언제 맞히는가"가 "얼마나 자주 맞히는가"보다 중요한 도메인에 적합한 접근

### 2-2. AlphaZero 기법의 경제 도메인 전이

AlphaGo Zero가 바둑에서 자기 대전(self-play)으로 초인적 성능을 달성한 핵심 아이디어 — **모델 기반 계획(model-based planning)** — 을 가격 시장에 이식했다.

- **Representation Network h(s)**: 256차원 시장 상태 → 압축된 latent state (6.4M 파라미터)
- **World Model g(s,a)**: 행동을 취했을 때 시장이 어떻게 변할지 학습 (6.5M 파라미터)
- **Prediction Network f(s)**: 현재 상태에서 최선의 행동과 가치를 추정 (6.0M 파라미터)
- **Action Model a(z)**: 맥락 인식 행동 사전 분포 학습 (~43K 파라미터)
- **MCTS**: 학습된 세계 모델 위에서 50회 시뮬레이션으로 최적 행동 탐색

단순 지도학습으로는 불가능한 **미래 시나리오 탐색 능력**을 확보한다는 점이 인상적이다.

### 2-8. World Model — 시장 물리학의 내면화

Dynamics Network `g(s_t, a_t) → (s_{t+1}, μ, σ²)`는 단순한 보조 네트워크가 아니라 이 시스템의 **핵심 세계 모델(World Model)** 이다.

MuZero에서 세계 모델의 역할은 환경 시뮬레이터를 대체하는 것이다. 바둑에서는 게임 규칙이 명확해 완벽한 시뮬레이터가 존재하지만, GPU 시장에는 공식 규칙이 없다. 이 프로젝트는 과거 가격 데이터에서 **시장의 인과 구조를 신경망으로 귀납**하여 그 공백을 채운다.

```
입력:  (현재 시장 latent s_t, 취할 행동 a_t)
출력:  (다음 시장 상태 s_{t+1},  보상 기댓값 μ,  보상 불확실성 σ²)
```

- 4개의 Transformer 블록 + Post-LN 잔차 연결: 행동-상태 상호작용의 복잡한 비선형성 학습
- **가우시안 분포 출력**: 점추정(μ)과 불확실성(σ²)을 동시에 출력하여 MCTS 탐색 중 신뢰 구간을 암묵적으로 반영
- 이 네트워크가 정확할수록 MCTS는 현실에 가까운 미래를 탐색하며, 정확도가 낮을수록 전체 시스템 신뢰도가 하락한다

세계 모델의 품질이 에이전트 성능의 상한선을 결정한다는 점에서, **World Model은 이 프로젝트의 가장 취약한 동시에 가장 중요한 구성 요소**다.

### 2-9. Action Model — 하드코딩 휴리스틱을 학습 가능한 사전 분포로 대체

`ActionModel`은 두 개의 서브모듈로 구성된 ~43K 파라미터의 경량 네트워크다.

**ActionEmbeddingLayer (임베딩 레이어)**

```
5D one-hot 행동 → 16D 학습 임베딩
```

표준 MuZero는 행동을 단순 one-hot 벡터로 취급한다. 이는 모든 행동이 동등하게 "다르다"고 간주하는 것이다. 반면 학습된 임베딩은 데이터로부터 **행동 간의 의미적 유사성**을 귀납할 수 있다:
- BUY_NOW와 WAIT_SHORT는 "매수 계열"로 가까운 임베딩을 학습할 수 있음
- SKIP은 "완전 포기"로 다른 행동들과 거리를 두는 임베딩을 학습할 수 있음

**ActionPriorNetwork (맥락 인식 사전 분포)**

```
256D latent state → 128D → 64D → 5D logits → π(a|s)
```

기존 코드에는 가격 추세에 따라 행동 가중치를 수동으로 조정하는 `utility_bias` 휴리스틱이 하드코딩되어 있었다:

```python
# 기존 하드코딩 방식
utility_bias[0] += 1.0 * trend_down - 1.2 * over_ma  # BUY_NOW
utility_bias[1] += 0.6 * over_ma + 0.5 * trend_up    # WAIT_SHORT
```

ActionPriorNetwork는 이 휴리스틱을 **데이터에서 end-to-end로 학습**한다. 시장 상태(latent state)를 입력받아 "이 상황에서 어떤 행동이 적합한가"의 확률 분포를 직접 출력한다.

최종 정책 칼리브레이션에서 0.15의 가중치로 반영된다:
```
final_π = 0.45 × MCTS + 0.25 × Reward + 0.15 × f-net + 0.15 × ActionModel
```

하드코딩된 인간 직관을 데이터 기반 학습으로 대체했다는 점에서, 이 모듈은 **시스템이 도메인 전문 지식 의존도를 낮추고 자기 적응력을 높이는 방향**으로 진화했음을 보여준다.

### 2-3. 불확실성의 명시적 모델링

보상 헤드가 점추정(μ)만이 아닌 **가우시안 분포 (μ, σ²)** 를 출력한다.

- `reward_logvar` 헤드가 Gaussian NLL 손실로 학습되어 예측 불확실성을 동시에 학습
- 신뢰도가 낮은 상황 (entropy > 1.58, confidence < 0.25)에서 safe_mode 발동
- 금융·의사결정 AI에서 점추정만으로는 충분하지 않다는 올바른 설계 판단

### 2-4. 완전 자동화된 온라인 학습 루프

```
매일 자정 → 수집 → 특징 추출 → 준비도 체크 → (30일 이후) 자동 재학습 → 품질 게이트 → 배포 판정
```

사람의 개입 없이 데이터 수집부터 릴리즈 판정까지 자동화되어 있다. 특히:

- 7일마다 재학습으로 **시장 드리프트 적응**
- 릴리즈 전 7개 품질 게이트 자동 검사로 **나쁜 모델의 자동 배포 차단**
- 결과를 한국어 Markdown 보고서로 자동 생성해 가독성 확보

이는 MLOps 관점에서도 견고한 설계다.

### 2-5. 결측 데이터의 엔지니어링적 처리

학습 데이터가 항상 완전하지 않다는 현실을 인정하고, `value=0.0 + missing_mask=0.0` 패턴으로 **결측 자체를 특징으로 인코딩**했다.

- 기술 지표 106차원 = 값 53 + 결측 마스크 53
- 데이터가 없을 때 임의 상수(예: 평균값)로 채우지 않고 "모른다는 사실"을 명시
- 모델이 데이터 품질에 따라 다른 행동을 학습할 수 있는 기반 제공

### 2-6. 경량 아키텍처의 실용성

약 19M 파라미터(h + g + f + a 합산)로 Apple MPS에서 실시간 추론이 가능하다.

- GPT급 모델이 아닌, 도메인에 맞게 최소화된 크기
- 체크포인트 단 2개 파일(~144MB)로 전체 모델 관리
- 클라우드 없이 개인 Mac에서 완전 운영 가능

### 2-7. 다층 칼리브레이션으로 모드 붕괴 방지

최종 행동 확률을 단일 소스에 의존하지 않고 4개 소스를 혼합한다:

```
final_π = 0.45 × MCTS + 0.25 × Reward + 0.15 × f-net + 0.15 × ActionModel
```

- 특정 행동 쏠림(mode collapse) 방지에 실증적으로 효과적
- ActionModel은 과거 데이터의 실제 행동 분포를 사전(prior)으로 제공

## 3. 단점 및 한계

### 3-1. 심각한 데이터 희소성

| 항목 | 수치 |
|------|------|
| 하루 수집 샘플 | 24개 GPU 모델 × 1일 = 24개 transition |
| 30일 기준 총 학습 샘플 | **631개** (2026-03-22 실측, 연속 날짜 쌍 기준) |
| 모델 파라미터 수 | ~19,000,000 |

파라미터 수 대비 학습 데이터가 극단적으로 부족하다.
배치 크기 32, 500 스텝으로 학습하면 동일 데이터가 **20회 이상 재사용**되어 과적합(overfitting) 위험이 상존한다.
실제로 품질 게이트를 통과하더라도 이것이 진짜 일반화 능력인지, 훈련 데이터 암기인지 구분하기 어렵다.

### 3-2. 단일 소스 의존 — Danawa

국내 가격 데이터가 **다나와 단일 크롤링**에 의존한다.

- 다나와 UI 변경 → 크롤러 즉시 동작 불능
- IP 차단, robots.txt 정책 변경 시 데이터 수집 중단
- 해외 가격(Amazon, Newegg)과 국내 가격의 괴리를 반영하지 않음
- 공식 제조사 판가, 쿠팡 등 대형 플랫폼 가격 미포함

데이터 소스가 다양할수록 모델의 견고성이 높아지지만, 현재는 SPOF(단일 장애점) 구조다.

### 3-3. 보상 함수의 과도한 단순화

```python
if action == "BUY_NOW":
    reward = pct_change          # 다음 날 가격이 오르면 좋음
if action in {"WAIT_SHORT", "WAIT_LONG"}:
    reward = -pct_change         # 다음 날 가격이 내리면 좋음
```

현실의 GPU 구매 결정에는 다음 요소들이 빠져 있다:

- **수수료 및 배송비**: 실제 구매 비용
- **기회 비용**: 관망하는 동안 가격이 급등할 때의 손실
- **기간 리스크**: WAIT_SHORT(수일)와 WAIT_LONG(수주)의 명확한 경계 없음
- **재고 고갈 리스크**: "기다리다가 물건이 없어지는" 시나리오
- **신제품 출시 이벤트**: 불연속적 가격 변화

보상 함수가 현실을 충분히 반영하지 못하면, 에이전트가 최적화하는 목표와 실제 사용자 이익이 어긋날 수 있다.

### 3-4. macOS 및 Apple MPS 종속성

전체 시스템이 사실상 **macOS + Apple Silicon** 환경에 종속되어 있다.

- `LaunchAgent`: macOS 전용 자동화 도구
- `torch.backends.mps`: Apple MPS 전용 GPU 가속
- `pmset`: macOS 전용 전원 관리 명령

Linux 서버나 클라우드 환경으로 이전하려면 LaunchAgent → systemd/cron, MPS → CUDA/CPU 전환이 모두 필요하다.

### 3-5. Cold Start 문제 — 30일 대기

모델 학습이 활성화되기까지 **30일의 데이터 축적이 필수**다.

> ✅ **2026-03-22 완료**: 30일 Cold Start 구간을 실제로 통과. 이 분석은 초기 구축 단계에서의 한계를 기술한 것이며, 현재는 학습·릴리즈·배포가 완료된 상태다. 이하 항목들은 서비스 재시작이나 새 GPU 모델 추가 시 여전히 적용되는 구조적 한계다.

- 초기 30일간은 추론 결과의 신뢰도가 사실상 무의미
- 서비스 재시작(데이터 초기화) 시 다시 30일 대기
- 새로운 GPU 모델이 시장에 진입할 때 해당 모델에 대한 30일 기다림 재발생

### 3-6. World Model의 근본 한계 — 과거로 미래를 시뮬레이션

World Model(Dynamics Network)은 이 시스템 전체의 **가장 취약한 연결 고리**다.

이 모델은 과거 30일의 가격 데이터로 학습된다. 즉 MCTS가 수행하는 50회 시뮬레이션은 사실상 **"과거에서 관찰된 패턴의 외삽"** 에 불과하다. 따라서 학습 분포 밖의 이벤트에는 근본적으로 대응할 수 없다:

- RTX 50 시리즈 출시 같은 **불연속적 가격 점프**: 학습 데이터에 없는 transition을 세계 모델이 전혀 예측 못 함
- 암호화폐 시장 급등으로 인한 GPU 수요 폭발: 뉴스 특징은 있지만 과거 유사 이벤트 데이터가 없으면 무의미
- 글로벌 반도체 공급망 위기: 장기 추세 변화를 30일 윈도우로 포착 불가

비유하자면, 이 세계 모델은 "맑은 날"의 시뮬레이터다. "폭풍"을 본 적이 없기 때문에 폭풍을 예측할 수 없다.

### 3-7. MCTS의 구조적 약점 — 얕은 탐색

```
50 simulations × 5 rollout steps = 250번의 dynamics 호출
```

바둑 AlphaZero가 수천~수만 시뮬레이션을 수행하는 것과 달리, 50회 시뮬레이션은 검색 공간을 매우 얕게 탐색한다.
World Model의 오차가 5 스텝 롤아웃을 거치며 누적되면, 탐색 결과의 신뢰도가 급락할 수 있다.

### 3-8. Action Model의 훈련 신호 약점

ActionPriorNetwork는 과거 가격 변화로부터 역산된 **의사 레이블(pseudo-label)** 로 학습된다. 즉 "다음 날 가격이 내렸으면 어제 BUY_NOW는 나쁜 행동"이라는 단순 규칙으로 레이블을 생성한다.

- 이 레이블은 노이즈가 많고 부정확 (하루 단위 변화로 장기 가치 판단 불가)
- 클래스 불균형이 심함: 시장 횡보 구간에서 모든 레이블이 HOLD/SKIP으로 편향
- `class_weights`로 균형을 맞추지만, 이는 데이터 편향의 근본 해결책이 아님

결과적으로 ActionModel이 학습하는 "맥락 인식"이 실제 시장 인과관계를 반영하는지, 단순히 가격 추세의 상관관계를 암기하는지 구분이 어렵다.

### 3-9. 평가 지표와 실제 사용자 경험의 괴리

```
방향 정확도 >= 55% → 릴리즈 허용
```

모델이 55%의 확률로 "가격 방향"을 맞히더라도, 실제 사용자가 추천을 따랐을 때 수익이 발생하는지는 별개의 문제다.
백테스트 평가(`AgentEvaluator`)는 미래 데이터 유출(lookahead bias) 없이 구현되어 있지만, 거래 비용과 실행 슬리피지가 반영되지 않는다.

### 3-10. 뉴스 감성 분석의 한계

- **영문 뉴스만 수집**: GPU 관련 한국어 뉴스, 네이버 카페, 커뮤니티(ruliweb, 퀘이사존 등)가 누락
- **폴백 스코어러**: Transformer 모델 없이 룰 기반으로 동작할 때 정확도 급락
- **GPU 도메인 특화 미흡**: 범용 감성 분석 모델(DistilBERT SST-2)은 GPU 시장 전문 용어에 최적화되어 있지 않음

## 4. 시사하는 바

### 4-1. RL을 현실 도메인에 적용할 때 보상 함수 설계가 전부다

이 프로젝트의 가장 큰 교훈은 **보상 함수의 중요성**이다.
바둑에서는 "이겼는가/졌는가"라는 명확하고 지연되지 않는 보상이 있다.
그러나 GPU 구매에서는 "잘 산 것인가"의 기준이 관점에 따라 다르다.

- 1주 뒤 가격 기준인가, 1개월 뒤 기준인가?
- 내가 필요할 때 산 것이 최고의 결정인가?
- 기다리다 못 산 것이 손실인가?

보상 함수가 현실을 충분히 대리하지 못하면 에이전트는 **사용자의 실제 이익이 아닌 보상 함수의 최적화자**가 된다. 이는 Goodhart's Law의 전형적인 사례다.

### 4-2. 데이터 희소성 도메인에서 딥러닝 모델의 적합성 재고

수십만 행의 학습 데이터가 있는 도메인에서는 딥러닝이 압도적이다.
그러나 **하루 24개 샘플**, **30일 = 720개 샘플** 수준에서는 다음 질문이 필요하다:

- XGBoost, LightGBM 등 전통 ML 방법론과의 비교가 있었는가?
- ARIMA, Prophet 같은 시계열 특화 방법과 비교했을 때 실제로 더 나은가?
- 19M 파라미터 신경망이 720개 샘플로 학습 가능한가?

복잡한 모델이 항상 좋은 것은 아니며, 데이터 양에 맞는 모델 복잡도 선택이 중요하다.

### 4-3. 개인용 자동화 AI 시스템의 현실적 가능성

이 프로젝트는 클라우드나 대규모 인프라 없이 **개인 Mac 한 대에서 완전히 동작하는 AI 시스템**을 구현했다.

- 프라이버시: 모든 데이터가 로컬 저장
- 비용: GPU 클라우드 비용 제로
- 제어권: 완전한 소유권

이는 "AI = 대기업 + 클라우드"라는 통념에 반하는 사례다.
LLM이 아닌 **도메인 특화 소형 모델**이 개인 기기에서 충분히 실용적임을 보여준다.

### 4-4. 릴리즈 파이프라인 = MLOps의 핵심

코드의 CI/CD처럼, ML 모델에도 **자동화된 품질 검증 게이트**가 필수임을 이 프로젝트가 잘 보여준다.

```
학습 완료 ≠ 배포 가능
```

7개의 품질 게이트가 자동 동작하여 나쁜 모델이 프로덕션에 진입하는 것을 방지한다.
특히 `mode_collapse` 감지와 `action_entropy` 모니터링은 RL 에이전트 특유의 퇴행(regression) 패턴을 잡아내는 중요한 장치다.

이 접근법은 **ML 모델 관리 시스템(Model Registry + 자동 검증)**의 축소판이며, 팀 단위 MLOps 시스템 설계에도 동일한 원칙이 적용된다.

### 4-5. 연속 학습(Continual Learning)의 실용적 과제

이 프로젝트는 7일마다 재학습하는 연속 학습 구조를 채택했다.
이 구조가 드러내는 연속 학습의 근본 과제들:

- **Catastrophic Forgetting**: 새 데이터로 학습하면 과거에 학습한 패턴이 덮어씌워질 수 있음
- **Distribution Shift**: 시장 구조 자체가 변하면(예: 새 세대 GPU 출시) 과거 데이터가 오히려 해가 됨
- **Evaluation의 어려움**: 어제 모델이 좋았던 이유가 오늘도 유효한지 판단 어려움

재학습 간격(7일), 학습 데이터 윈도우(30일), 게이트 임계값은 모두 경험적으로 설정된 값이다.
이 하이퍼파라미터들의 민감도 분석이 없는 상태에서는 "잘 동작하는 이유"가 명확하지 않다.

### 4-6. 해석 가능성(Explainability)의 부재

사용자가 "RTX 5060을 지금 사세요"라는 추천을 받았을 때, **왜 그 결론에 도달했는지**를 설명할 수 없다.

- 어떤 특징이 이 결정에 가장 크게 기여했는가? (Feature importance 없음)
- MCTS가 어떤 미래 시나리오를 상상했는가? (시뮬레이션 경로 불투명)
- 이 추천을 신뢰할 수 있는 근거가 confidence 수치 이외에 있는가?

금융·구매 의사결정 AI에서 해석 가능성은 신뢰 확보와 오류 디버깅 양쪽에서 모두 중요하다.

### 4-7. World Model 품질이 전체 시스템의 상한선

이 프로젝트가 보여주는 중요한 시사점은 **모델 기반 강화학습에서 세계 모델의 오차는 전파된다**는 것이다.

MCTS 탐색은 세계 모델이 정확하다는 전제에서 최적 행동을 찾는다. 세계 모델이 부정확하면 MCTS는 "환상의 미래"를 탐색하는 것이 된다. 이 오차 전파 구조는 다음을 의미한다:

- Prediction Network f(s)가 아무리 좋아도 g(s,a)가 나쁘면 탐색 결과 전체가 신뢰 불가
- 데이터 희소 도메인에서 세계 모델 학습은 특히 위험 — 모델이 자신의 편향을 "시장의 법칙"으로 학습할 수 있음
- **세계 모델 검증**이 에이전트 성능 검증만큼 중요하지만, 현재 파이프라인에는 g(s,a)의 예측 정확도를 독립적으로 측정하는 게이트가 없음

세계 모델의 예측 오차(next state MSE, reward calibration error)를 별도 품질 게이트로 모니터링하는 것이 시스템 안정성에 크게 기여할 것이다.

### 4-8. "작동하는 프로토타입"과 "검증된 시스템"의 차이

이 프로젝트는 기술적으로 완성도 높은 프로토타입이다.
그러나 다음 질문에 답하지 못하면 **검증된 시스템**이라고 부르기 어렵다:

1. 이 에이전트를 6개월 실제 사용했을 때 GPU를 적절한 타이밍에 구매했는가?
2. 추천을 따르지 않았을 때보다 따랐을 때 실제로 돈을 절약했는가?
3. 무작위 전략(Random) 대비 통계적으로 유의미한 성능 차이가 있는가?

백테스트는 과거 데이터를 사용하기 때문에 낙관적으로 편향될 수 있다.
실제 가치는 **Live A/B 테스트** 또는 **실제 구매 기록 추적**을 통해서만 증명된다.
